## P1 — 高优先级待修

### P1-1 录屏选区在混合 DPI 多屏下坐标/尺寸错位

- **现象**：主屏 150% + 副屏 100% 等多 DPI 显示器时，录屏（`recorder-select` / `recording-border`）的选区坐标或尺寸按错误 DPI 缩放，选区与鼠标实际位置偏移。
- **证据**：
  - `src-tauri/src/services/recording_service.rs:1372 / 1396 / 1576` 三处均用 `let dpi = GetDpiForSystem(); ... dpi as f64 / 96.0` —— **`GetDpiForSystem()` 返回系统全局 DPI，不是所选窗口所在显示器的 DPI**。
  - 对照 `src-tauri/src/screenshot.rs:121 / 659` 截图侧用 `EnumDisplayMonitors` + `MONITORINFO`（**per-monitor**，正确）。两模块 DPI 取法不一致。
- **影响**：截图正确、录屏错误；仅在混合 DPI 多屏环境触发，单屏/同 DPI 无感——故长期"偶发"难定位。
- **修复建议**：录屏改用 `MonitorFromWindow(hwnd, MONITOR_DEFAULTTONEAREST)` + `GetDpiForMonitor()` 取**窗口所在显示器** DPI；或在 `window_at_point` 命中后把该窗的 monitor DPI 一并回传前端。与 screenshot.rs 的 per-monitor 取法对齐。
- **严重度**：P1（功能正确性，特定环境必现）

### P1-2 ffmpeg 未纳入打包资源 → 打包后录屏/转码保存失败

- **现象**：`pnpm tauri build` 产物里录屏录完点保存无文件、薄荷音频转 MIDI 失败。
- **证据**：
  - `src-tauri/tauri.conf.json:44` `bundle.resources` 仅 `"../bundled-dlc": "bundled-dlc/"`，**不含 `../external-deps`**。
  - `src-tauri/src/services/recording_service.rs` 优先取 `external-deps/全局/ffmpeg/ffmpeg.exe`，回退系统 PATH。
  - 设计意图是 `prepare-bundled-dlc.mjs` 把 ffmpeg 打成 `.mujin` 经 `bundled-dlc` 解压到 `user_external_deps/全局/`。**需确认 `.mujin` 实际生成且解压链路完整**——当前 `external-deps/全局/ffmpeg/ffmpeg.exe` 在 dev 下存在，但打包资源未显式纳入，依赖 `.mujin` 链路不能断。
- **影响**：dev 模式正常（直接读 external-deps），**打包（release）必现**保存失败。
- **修复建议**：① 跑一次 `node scripts/prepare-bundled-dlc.mjs` 确认产出含 ffmpeg 的 `.mujin`；② 若 `.mujin` 链路不稳，直接在 `tauri.conf.json` 的 `bundle.resources` 追加 `"../external-deps": "external-deps/"` 兜底；③ 在 recording_service 解压失败时给出明确前端提示（已有「请确认 external-deps/全局/ffmpeg 可用」文案，需保证路径正确）。
- **严重度**：P1（打包阻断，release 必现）

## P2 — 中低优先级（加固 / 边界 / 缺失能力）

### P2-1 `.mujin` / `.mufurong` 无签名/哈希校验 → 沙箱逃逸与本地提权风险

- **现象**：任意写入 `user_plugins/` 的 `.mujin` 包会被自动解压并执行其 `index.js`（IIFE）。
- **证据**：
  - `src/core/settings/DevConsole.tsx:283-298` 安装 `.mufurong`/`.mujin` 直接写 `user_plugins/<fileName>`，由 `extract_mufurong_plugins` / `extract_mujin_deps` 自动扫描解压。
  - `src-tauri/src/main.rs:310-331` 从 `bundled-dlc/` 复制 `.mufurong`/`.mujin` 到用户目录。
  - 插件 IIFE 可访问 `window.__HOST_API__`（含 `createFloatingWindow` / `invoke` 任意命令 / `emit`），权限等同宿主。
  - 全仓 grep `mujin|signature|integrity|sha256|校验` 在 src-tauri 仅命中 `.mujin` 复制与 `clip_fast_image_hash`（功能 hash，非安全校验），**未发现对包本身的签名/完整性校验**。
- **影响**：本地桌面应用 + 用户主动安装包场景下威胁中等（非 web 远程攻击面）；但若用户从不可信渠道下载 `.mujin`，可在本机以应用权限执行任意代码。
- **修复建议**：加载 `.mujin` 前做 `mtime` + `sha256` 校验（或内嵌签名），拒绝校验失败/未知来源的包；DevConsole 安装时显式提示来源风险。
- **严重度**：P2（安全加固，非紧急功能 bug）

### P2-2 桌宠多显示器仅覆盖主屏

- **现象**：`DeskpetPet.tsx` 用 `screen` 主屏尺寸全屏化 + 沿主屏底部漫游，多屏时副屏无宠物、且主屏全屏透明窗可能遮挡点击穿透范围判断。
- **证据**：`src/components/DeskpetPet.tsx` 挂载即 `setFullscreen(true)` + 主屏 `screen` 尺寸。
- **修复建议**（延后，MVP 可接受）：改用 `screen.getMonitors()` 选定活动显示器，或为每个显示器创建独立宠物实例；当前 MVP 已用 `setIgnoreCursorEvents(true)` 点击穿透，不阻断操作。
- **严重度**：P2（体验边界，多屏用户）

### P2-3 桌宠真实光标跟随（rdev 全局钩子）未实现

- **现象**：当前桌宠为自走动画小球，不跟随真实鼠标。
- **证据**：`src-tauri/Cargo.toml` 无 `rdev`；`src-tauri/src/services/device.rs` 不存在。参考实现 `got-it/BongoCat-master`（forked kunkunsh/rdev，MIT）。
- **影响**：功能缺失，非 bug；属 MVP 之后能力。
- **修复建议**（下一阶段）：加 `device.rs` + Cargo 依赖（注意与 `windows` 0.62 / `windows-capture` 2.0 共存），经 Tauri 事件把光标坐标推给桌宠浮窗。
- **严重度**：P2（待开发能力）
