## 一、核心需求拆解

| 需求 | 技术含义 | 设计约束 |
|------|---------|---------|
| AI时刻处于主动攻防状态 | 主循环持续运行，非请求-响应模式 | 必须是事件驱动的长驻进程 |
| 用户可随时输入提示词 | 异步指令注入，不阻塞主循环 | 指令队列 + 状态快照机制 |
| 当前执行不中断 | 主循环原子性，指令在步骤间插入 | 步骤级检查点，不可抢占 |
| 除非强制中断切换 | 支持优先级抢占和状态保存 | 信号机制 + 上下文持久化 |
| 攻防状态始终维持 | 状态机永不退出，故障自恢复 | 守护进程 + 崩溃重启 + 状态恢复 |

```

                       │
                       ▼

│                   指令调度器（Dispatcher）                  │
│                                                          │
│  职责：                                                   │
│    1. 监听指令队列                                        │

│    3. 按优先级决定：注入/抢占/排队                         │

│                                                          │
│  while True:                                             │
│    state = load_state()           # 加载当前攻防状态       │

│    │                                                     │
│    │ ←── 检查点：此处检查指令队列 ───┐                    │
```

## 三、状态机设计

### 3.1 攻防状态定义

```
EngineState = {
    mode: OFFENSIVE | DEFENSIVE | RECON | MONITORING | STANDBY,
    phase: RECON → SCAN → ANALYZE → EXPLOIT/DEFEND → PIVOT → CLEANUP → REPORT,
    target: {url, ip, scope, rules_of_engagement},
    context: {
        discovered_assets: [...],      # 已发现资产
        vulnerabilities: [...],        # 已发现漏洞
        credentials: [...],            # 已获取凭证
        sessions: [...],               # 活跃会话
        fingerprints: [...],           # 目标指纹
    },
    history: [                         # 执行历史
        {step_id, action, result, timestamp, ai_decision}
    ],
    user_directives: [                 # 待执行用户指令
        {id, type, content, priority, timestamp}
    ],
    metrics: {
        requests_sent, responses_received, errors, duration
    }
}
```

### 3.2 状态转换矩阵

| 当前状态 | 触发条件 | 目标状态 | AI行为 |
|---------|---------|---------|--------|
| STANDBY | 启动/用户指定目标 | RECON | AI规划侦察策略 |
| RECON | 侦察完成 | SCAN | AI分析侦察结果，规划扫描 |
| SCAN | 扫描完成 | ANALYZE | AI评估发现，规划深度分析 |
| ANALYZE | 分析完成 | OFFENSIVE/DEFENSIVE | AI决策攻防方向 |
| OFFENSIVE | 攻击目标达成/用户切换 | PIVOT/DEFENSIVE | AI规划下一步行动 |
| DEFENSIVE | 防御部署完成 | MONITORING | AI持续监控威胁 |
| MONITORING | 检测到威胁 | DEFENSIVE | AI自动响应 |
| 任意状态 | 用户[IMMEDIATE]指令 | 用户指定 | 保存当前状态，切换 |
| 任意状态 | 异常/崩溃 | STANDBY | 守护进程重启，恢复状态 |

### 3.3 步骤级原子性保证

```
Step生命周期：
  BEGIN_STEP
    ├─ pre_check:    验证前置条件（目标可达、权限有效）
    ├─ ai_plan:      AI规划具体操作（L0/L1/L2级联）
    ├─ execute:      执行操作（网络请求、扫描、利用）
    ├─ post_verify:  验证执行结果
    └─ ai_reflect:   AI反思，更新状态
  END_STEP
  │
  ▼
  CHECKPOINT: 检查指令队列  ← 唯一可注入用户指令的点
```

**关键约束**：用户指令只能在`CHECKPOINT`处注入，保证每个步骤的原子性。`[IMMEDIATE]`类型指令是唯一例外——通过信号机制强制中断当前步骤，但会先保存状态快照。

## 四、AI主动攻防机制

### 4.1 AI决策循环（OODA Loop）

| 阶段 | AI职责 | 模型 | 延迟 | 输出 |
|------|--------|------|------|------|
| Observe（观察） | 采集目标响应、网络流量、系统状态 | 传感器+规则 | <10ms | 原始情报 |
| Orient（判断） | 评估态势、识别模式、关联历史 | L1 Gemma3-4B | <100ms | 态势评估报告 |
| Decide（决策） | 选择最优行动方案、评估风险 | L2 Qwen3-32B | <5s | 行动计划 |
| Act（行动） | 执行攻防操作、采集结果 | 执行器 | varies | 执行结果 |

**循环不中断**：OODA循环持续运行，即使用户输入指令，当前循环完成后才注入。

### 4.2 主动攻防策略生成

```
输入：
  - 当前状态（已发现资产、漏洞、会话）
  - 目标范围（rules of engagement）
  - 历史执行记录
  - 知识库（RAG检索相似场景）

AI推理（Qwen3-32B）：
  Prompt: "
  你是攻防引擎的决策核心。当前状态：
  - 已发现资产：{discovered_assets}
  - 已发现漏洞：{vulnerabilities}
  - 活跃会话：{sessions}
  - 历史动作：{recent_history}
  - 目标范围：{scope}
  - 知识库参考：{rag_results}
  
  基于以上信息，规划下一步行动（仅1步）：
  1. 行动类型（recon/scan/exploit/defend/monitor/pivot）
  2. 具体操作
  3. 预期结果
  4. 风险评估
  5. 回退方案
  "

输出（结构化JSON）：
  {
    "action_type": "exploit",
    "target": "https://target.com/api/v1/upload",
    "technique": "CVE-2024-XXXX",
    "payload": "...",
    "expected_result": "获取RCE",
    "risk": "low",
    "fallback": "若失败则尝试CVE-2024-YYYY"
  }
```

### 4.3 持续学习与策略进化

| 机制 | 频率 | 方法 | 效果 |

| 短期学习 | 每步 | Reflector更新状态和上下文 | 避免重复失败 |
| 中期学习 | 每轮 | GEPA优化Prompt | 提升决策准确率 |

## 五、用户指令异步注入机制

### 5.1 指令类型与优先级

| 类型 | 优先级 | 行为 | 示例 |
|------|--------|------|------|
| IMMEDIATE | P0（最高） | 信号中断当前步骤，保存快照，立即切换 | "停止当前扫描，转向192.168.1.0/24" |
| NEXT_STEP | P1 | 当前步骤完成后，下一轮优先执行 | "下一步尝试SQL注入" |
| APPEND | P2 | 加入任务池，按调度策略执行 | "记录所有发现的子域名" |
| MODIFY | P1 | 修改当前策略参数，不中断 | "提高扫描并发到50" |

### 5.2 指令队列与检查点机制

```
                    指令队列（Redis Sorted Set, score=priority）
                    ┌─────────────────────────────────┐
                    │  P0: [IMMEDIATE] 切换目标       │
用户输入 ────────►  │  P1: [NEXT_STEP] 尝试SQLi       │
                    │  P1: [MODIFY] 提高并发           │
                    │  P2: [APPEND] 记录子域名         │
                    └──────────────┬──────────────────┘
                                   │
                                   ▼
    主循环检查点（每步骤间隙）
    ┌──────────────────────────────────────────┐
    │                                          │
    │  def checkpoint():                       │
    │    # 1. 检查P0（IMMEDIATE）              │
    │    if has_immediate():                   │
    │      snapshot = save_current_state()     │
    │      execute_immediate()                 │
    │      return SWITCH                       │
    │                                          │
    │    # 2. 检查P1（NEXT_STEP/MODIFY）       │
    │    if has_next_step():                   │
    │      inject_into_planner()               │
    │      return INJECT                       │
    │                                          │
    │    # 3. 检查P2（APPEND）                 │
    │    drain_append_to_task_pool()           │
    │                                          │
    │    return CONTINUE                       │
    └──────────────────────────────────────────┘
```

### 5.3 上下文保存与恢复

```
快照内容（保存到Redis/磁盘）：
{
    "snapshot_id": "uuid",
    "timestamp": "2026-07-16T10:30:00Z",
    "engine_state": {完整EngineState},
    "current_step": {
        "step_id": "step_42",
        "phase": "execute",
        "partial_result": {...},
        "interruptible": true
    },
    "reason": "USER_IMMEDIATE_SWITCH",
    "resumable": true
}

恢复时：
  1. 加载快照
  2. 检查current_step.phase
  3. 若phase=execute且不可恢复 → 重新执行整个step
  4. 若phase=pre_check/ai_plan → 从该阶段继续
  5. 恢复engine_state
```

## 六、稳定性保障机制

### 6.1 永不中断的守护架构

```
┌─────────────────────────────────────────────────┐
│              守护进程（Supervisor）                │
│                                                 │
│  职责：                                         │
│    - 监控主循环心跳（每5秒一次）                  │
│    - 主循环崩溃时自动重启                        │
│    - 重启时从最近快照恢复状态                    │
│    - 记录崩溃日志供AI分析                        │
└──────────────────┬──────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────────┐
│              主攻防循环（Main Loop）               │
│                                                 │
│  心跳：每5秒写入 Redis key=heartbeat             │
│  状态：每步完成后持久化到 Redis + 磁盘            │
│  日志：全量操作日志到 ClickHouse                 │
└──────────────────┬──────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────────┐
│              故障恢复流程                         │
│                                                 │
│  1. 守护进程检测到心跳超时（>15秒）              │
│  2. 收集崩溃core dump + 日志                    │
│  3. AI分析崩溃原因（L2推理）                     │
│  4. 从最近快照恢复状态                           │
│  5. 调整可能导致崩溃的参数                       │
│  6. 重启主循环，恢复攻防状态                     │
│  7. 通知用户：崩溃已恢复，原因：{AI分析}         │
└─────────────────────────────────────────────────┘
```

### 6.2 各层容错设计

| 层级 | 故障场景 | 处理策略 | 恢复时间 |
|------|---------|---------|---------|
| 网络层 | 目标不可达 | 自动切换代理/重试3次 | <30s |
| 执行层 | 工具调用超时 | 超时取消+降级方案 | <10s |
| AI层 | LLM推理失败 | L2→L1→L0逐级降级 | <200ms |
| AI层 | LLM输出格式错误 | Schema校验+重试+修复Prompt | <5s |
| 状态层 | 状态序列化失败 | 回退到上一个有效快照 | <5s |
| 进程层 | 主循环崩溃 | 守护进程重启+状态恢复 | <30s |
| 硬件层 | GPU故障 | 切换到CPU推理或备用GPU | <60s |

### 6.3 资源隔离与背压

```
资源池分配：
  ├─ AI推理资源（GPU）：独立线程池，不与执行层竞争
  ├─ 网络请求资源：连接池+信号量，防止耗尽FD
  ├─ CPU密集任务：进程池隔离，防止GIL阻塞
  └─ 内存：每步操作内存限制，超限自动GC

背压机制：
  当某层过载时：
  ├─ 网络层过载 → 降低请求频率，增加延迟抖动
  ├─ AI层过载 → 降级到L0规则引擎，排队等待L1/L2
  └─ 执行层过载 → 任务队列排队，优先级调度
```

## 八、技术栈与实现细节

### 8.1 核心技术栈

| 组件 | 选型 | 理由 |
|------|------|------|
| 主循环引擎 | Python asyncio | 异步I/O，适合网络密集型攻防 |
| 状态持久化 | Redis + SQLite | Redis热数据，SQLite冷备份 |
| 指令队列 | Redis Sorted Set | 优先级排序，原子操作 |
| 守护进程 | systemd + supervisor | Linux原生，可靠 |
| AI推理 | vLLM（本地） | 高吞吐，连续批处理 |
| AI编排 | LangGraph | 状态机管理，条件路由 |
| 知识库 | Milvus + Neo4j | 向量检索+图关联 |
| 可观测性 | OpenTelemetry + ClickHouse | 全链路trace |
| 工具执行 | 子进程隔离 | 防止工具崩溃影响主循环 |

### 8.2 主循环核心代码结构

```python
class AttackDefenseEngine:
    """持续运行的攻防引擎主循环"""
    
    async def run_forever(self):
        """主循环 — 永不退出"""
        while True:
            try:
                # 1. 加载状态
                state = await self.load_state()
                
                # 2. 检查点：处理用户指令
                state = await self.checkpoint_handle_directives(state)
                
                # 3. OODA: Observe
                observation = await self.observe(state)
                
                # 4. OODA: Orient (L1快速评估)
                assessment = await self.orient_l1(state, observation)
                
                # 5. OODA: Decide (L0/L1/L2级联决策)
                plan = await self.decide_cascaded(state, assessment)
                
                # 6. 检查点：执行前再次检查IMMEDIATE指令
                if await self.has_immediate_directive():
                    state = await self.handle_immediate(state)
                    continue
                
                # 7. OODA: Act (原子执行)
                result = await self.act_atomic(plan)
                
                # 8. OODA: Reflect (AI反思，更新状态)
                state = await self.reflect_l2(state, result)
                
                # 9. 持久化状态
                await self.save_state(state)
                
                # 10. 心跳
                await self.heartbeat()
                
            except Exception as e:
                # 异常不退出，记录后继续
                await self.handle_error(e)
                await asyncio.sleep(1)  # 防止异常风暴
    
    async def checkpoint_handle_directives(self, state):
        """检查点：处理指令队列"""
        # P0 IMMEDIATE
        immediate = await self.queue.pop_immediate()
        if immediate:
            await self.snapshot_save(state)
            return await self.execute_immediate(state, immediate)
        
        # P1 NEXT_STEP
        next_step = await self.queue.pop_next_step()
        if next_step:
            state.user_context = next_step.parsed_directive
        
        # P1 MODIFY
        modifies = await self.queue.pop_all_modify()
        for mod in modifies:
            state = await self.apply_modify(state, mod)
        
        # P2 APPEND
        appends = await self.queue.pop_all_append()
        state.task_pool.extend(appends)
        
        return state
    
    async def decide_cascaded(self, state, assessment):
        """三级级联决策"""
        # L0: 规则匹配
        plan = self.rules_engine.match(state, assessment)
        if plan and plan.confidence > 0.9:
            return plan
        
        # L1: 轻量LLM
        plan = await self.l1_llm.decide(state, assessment)
        if plan.confidence > 0.7:
            return plan
        
        # L2: 重量LLM + RAG
        rag_context = await self.rag.retrieve(state, assessment)
        plan = await self.l2_llm.decide(state, assessment, rag_context)
        
        # 回写L0规则缓存
        if plan.confidence > 0.8:
            self.rules_engine.add_pattern(state, assessment, plan)
        
        return plan
```

## 十、关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 并发模型 | asyncio单线程+进程池 | I/O密集型，避免GIL问题 |
| 状态持久化 | 每步保存，Redis+SQLite双写 | 崩溃后最多丢失1步 |
| 指令注入点 | 步骤间检查点 | 保证步骤原子性 |
| 强制中断 | 信号+快照保存 | 可恢复，不丢状态 |
| AI降级 | L2→L1→L0三级 | 保证决策不中断 |
| 错误处理 | 不退出，记录后继续 | 攻防状态始终维持 |
| 上下文管理 | 滑动窗口+L1/L2摘要压缩 | 防止LLM窗口溢出 |
| 守护进程 | systemd | Linux原生，最可靠 |

**核心设计哲学**：主循环是一个永不退出的while True，任何异常都被捕获处理，任何用户指令都在检查点注入，任何崩溃都由守护进程恢复。攻防状态是一个持久化的状态机，存在Redis里，即使进程重启也能恢复到上一步。AI是嵌入决策链的智能核心，三级级联保证即使GPU故障也有规则兜底。
