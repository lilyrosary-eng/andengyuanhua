# AI增强与AI辅助 — 实现方案与落地姿势

> 防御性安全基础设施框架 · AI核心层设计
> 日期：2026-07-17

---

## 一、概念区分

| 维度 | AI增强（AI Enhancement） | AI辅助（AI Assistance） |
|------|------------------------|------------------------|
| **定位** | AI嵌入系统决策链路，系统自主运行 | AI作为工具，人类/系统按需调用 |
| **触发** | 持续运行，事件驱动 | 按需触发，请求-响应 |
| **决策权** | AI拥有部分自主决策权 | AI提供建议，人类/系统保留决策权 |
| **延迟要求** | 实时（<200ms）到准实时（<5s） | 可容忍较高延迟（<30s） |
| **失败影响** | 影响系统核心行为 | 仅影响单次建议质量 |
| **典型场景** | 自适应限流、指纹策略选择、异常检测 | 代码生成、漏洞分析报告、协议推断 |

两者不互斥，而是分层共存：AI增强在底层持续运行，AI辅助在上层按需介入。

                       │ 上下文传递

│                   AI增强层（持续）                     │
│  自适应策略 · 异常检测 · 模型路由 · 知识检索 · 反馈闭环   │

核心设计原则：**轻模型在底层做实时决策，重模型在上层做深度分析，规则引擎兜底**。

## 三、AI增强层实现方案

### 3.1 三级推理引擎

系统每个请求/事件需要经过AI决策，但不能让AI成为性能瓶颈。方案是**三级级联**：

| 级别 | 引擎 | 延迟 | 决策范围 | 触发条件 |
|------|------|------|---------|---------|
| L0 | 规则引擎（OPA/Rego） | <1ms | 已知模式的确定性决策 | 匹配规则库 |
| L1 | 轻量LLM（Gemma3-4B） | <100ms | 模式识别、分类、评分 | L0未命中 |
| L2 | 重量LLM（Qwen3-32B） | <5s | 复杂推理、策略生成、代码分析 | L1置信度<0.7 |

**级联逻辑**：

```
输入事件 → L0规则匹配
  ├─ 命中 → 直接执行（<1ms，覆盖80%请求）
  └─ 未命中 → L1轻量推理
      ├─ 置信度≥0.7 → 执行L1决策（<100ms，覆盖15%请求）
      └─ 置信度<0.7 → L2深度推理
          ├─ 生成决策 + 写入L0规则缓存（<5s，覆盖5%请求）
          └─ 异步执行
```

关键点：L2的决策结果会**回写L0规则缓存**，使系统在使用中持续加速——相同情况第二次出现时直接走L0。

### 3.2 模型路由（成本优化）

基于RouteLLM（ICLR 2025）的实践数据：用轻量路由器分类查询复杂度，简单查询走小模型，复杂查询走大模型，可削减75-85%成本，质量维持95%。

**路由策略矩阵**：

| 任务类型 | 首选模型 | Fallback | 路由依据 |
|---------|---------|----------|---------|
| 实时分类/评分 | Gemma3-4B-IT（本地） | 规则引擎 | 延迟<100ms |
| 代码生成 | DeepSeek-Coder-V2-16B | Qwen3-32B | 代码任务专用 |
| 漏洞推理 | Qwen3-32B-Instruct | Claude 3.5（云端） | 深度推理 |
| 视觉理解 | Qwen2.5-VL-7B | GPT-4V（云端） | 图片输入 |
| 策略优化 | Qwen3-32B + PPO Agent | 规则引擎 | 强化学习 |

**级联容错**：主模型故障时自动切换备用模型，每次fallback发出OpenTelemetry span属性记录。2026年4月OpenAI多小时故障中，有路由器的团队通过自动故障转移避免了全面中断。

### 3.3 本地LLM部署

**推理框架选型**（2025年8月实测，GPT-OSS-120B on 2x H100）：

| 指标 | vLLM | SGLang | TensorRT-LLM |
|------|------|--------|-------------|
| TTFT（并发1） | **0.053s** | 0.125s | 0.177s |
| 吞吐（并发100） | **4,741 tok/s** | 3,221 tok/s | 1,942 tok/s |
| 吞吐（并发50） | 2,211 tok/s | **3,108 tok/s** | 2,162 tok/s |
| 适用场景 | 通用高并发 | 结构化输出 | 极限硬件效率 |

选型结论：**vLLM作为通用推理引擎，SGLang用于Agent结构化输出场景**。

**量化方案**：

| 格式 | 70B模型大小 | 质量损失 | 适用场景 |
|------|-----------|---------|---------|
| BF16 | ~140 GB | 基准 | 无需量化 |
| FP8 | ~70 GB | 几乎无损 | H100/H200 |
| **AWQ INT4** | **~35 GB** | **1-3%** | **生产默认** |
| GPTQ INT4 | ~35 GB | 2-5% | AWQ不可用时备选 |

AWQ成为2026年默认INT4格式的原因：通过校准识别1%的"显著权重通道"进行保护，比GPTQ质量更高、速度更快。70B模型从需要2张A100变为单张A100 80G，吞吐提升约1.5倍。

**部署配置**：

```
本地LLM集群：
  推理引擎：vLLM（连续批处理 + PagedAttention）
  模型矩阵：
    - Gemma3-4B-IT（AWQ INT4, ~2.5GB显存）  → L1实时决策
    - Qwen3-32B-Instruct（AWQ INT4, ~18GB）  → L2深度推理
    - Qwen2.5-VL-7B（AWQ INT4, ~4.5GB）      → 视觉理解
    - DeepSeek-Coder-V2-16B（AWQ INT4, ~9GB） → 代码生成
  硬件需求：单卡A100 80G可全部部署，或4卡RTX 4090分布式
```

### 3.4 RAG知识检索

朴素RAG（嵌入→top-k→塞入prompt）会快速触顶。针对四种失败模式的解法：

| 失败模式 | 症状 | 对应技术 | 成本倍数 |
|---------|------|---------|---------|
| 词汇不匹配 | 查询jargon与文档措辞不匹配 | HyDE（假设文档嵌入） | 2x |
| 多跳推理 | 需要跨文档关联 | Query分解 | 3x |
| 全局问题 | 需要跨多文档聚合 | GraphRAG | varies |
| 开放探索 | 模型不确定该搜什么 | Agentic检索 | 8x |

**生产RAG管线**（成本路由策略）：

```
查询输入 → 查询分类器（Gemma3-4B）
  ├─ 简单事实查询 → Hybrid检索（BM25+向量）+ Rerank → 直接回答
  ├─ 复杂推理查询 → HyDE生成假设答案 → 检索 → Rerank → 回答
  ├─ 全局聚合查询 → GraphRAG社区摘要检索 → 回答
  └─ 开放探索查询 → Agentic检索（多轮搜索+反思） → 回答
```

实测效果：HotpotQA上Recall@5从38%（朴素）提升到56%（HyDE），GraphRAG在全局问题上EM准确率达64%。

### 3.5 自改进闭环

**闭环架构**：

```
执行任务 → 采集结果 → 效果评估
  ├─ 成功 → 记录成功经验 → 知识库更新
  └─ 失败 → 分析失败原因 → 策略调整 → 生成新策略 → 小规模测试
      ├─ 有效 → 知识库更新 + 模型微调
      └─ 无效 → 继续分析
```

**训练方法选型**：

| 方法 | 原理 | 资源需求 | 适用场景 |
|------|------|---------|---------|
| DPO | 闭式解偏好优化，无需RL循环 | 单卡80GB | 策略对齐 |
| 迭代DPO | 生成与奖励模型交替训练 | 单卡80GB | 在线学习 |
| RLAIF | 大模型生成偏好数据替代人工标注 | 中等 | 降低标注成本 |
| RLSF | 模型自身置信度作为奖励信号 | 低 | 无标注场景 |
| GEPA | 自然语言反思优化prompt | 极低（35x少于RL） | Prompt自动优化 |

**GEPA（2025年7月，DSPy）**是关键突破：用自然语言反思而非标量奖励优化prompt，比GRPO（RL方法）效果好10-20%，且使用35倍更少的rollout。实测效果：RAG任务从42%提升到61%（+19%），ReAct Agent从24%提升到51%（+27%）。

**强化学习管线**（以爬虫策略为例）：

```
State  = {指纹状态, 代理健康度, 目标响应码分布, 反爬强度评分}
Action = {选择指纹模板, 选择代理, 调整请求频率, 切换UA}
Reward = +10*(成功获取数据) -5*(触发CAPTCHA) -10*(IP被封) -1*(请求耗时)

算法：PPO（Proximal Policy Optimization）
训练频率：每周基于新采集数据微调一轮
评估指标：成功率、平均请求耗时、CAPTCHA触发率、IP封禁率
```

## 四、AI辅助层实现方案

### 4.1 Agent架构

采用**Plan-and-Execute + Reflection**混合模式：

```
用户请求/系统事件
  ↓
Planner（Qwen3-32B）：制定执行计划
  ↓
Executor（工具调用）：按计划逐步执行
  ├─ 工具1：代码分析器
  ├─ 工具2：漏洞扫描器
  ├─ 工具3：协议解析器
  └─ 工具N：...
  ↓
Reflector（Qwen3-32B）：评估执行结果
  ├─ 质量达标 → 输出结果
  └─ 质量不足 → 修改计划 → 重新执行
```

### 4.2 工具调用（Function Calling）

**五大生产原则**：

| 原则 | 内容 | 影响 |
|------|------|------|
| Schema设计 | 动词+名词语义化名称、适用/不适用场景、参数全称不缩写、enum值 | 决定80%调用准确率 |
| 并行调用 | 独立工具并行执行，asyncio.Semaphore控制并发 | 3个并行调用耗时=max而非sum |
| 智能路由 | L1闲聊直接回答、L2 embedding预选top-8工具、L3执行 | 工具>50个时防准确率下降 |
| 结构化返回 | status + summary + data三要素JSON | 减少Agent二次理解错误 |
| 错误处理 | 指数退避重试 + 超时控制 + 降级fallback + suggestion字段 | 工具失败≠Agent失败 |

**工具Schema示例**：

```json
{
  "name": "analyze_vulnerability",
  "description": "分析目标系统的潜在漏洞。适用于Web应用安全评估，不适用于二进制漏洞分析。",
  "parameters": {
    "target_url": {"type": "string", "description": "目标URL，包含协议和端口"},
    "tech_stack": {"type": "array", "items": {"type": "string"}, "description": "已识别的技术栈"},
    "depth": {"type": "string", "enum": ["quick", "standard", "deep"], "description": "扫描深度"}
  },
  "returns": {
    "status": "success|partial|error",
    "summary": "发现3个高危漏洞，2个中危漏洞",
    "data": {"vulnerabilities": [...]}
  }
}
```

### 4.3 Prompt工程工业化

**从手工艺到系统工程的转变**：

| 时代 | 方法 | 类比 |
|------|------|------|
| 2022-2023 | 手工Prompt工程 | 汇编语言 |
| 2024-2025 | DSPy + GEPA | 高级编程语言 |
| 2026+ | 语言自我反思优化 | 编译器自动化 |

**DSPy工作流**：

```python
import dspy

# 1. 定义Signature（声明输入/输出行为，不写prompt）
class VulnerabilityAnalysis(dspy.Signature):
    """分析目标系统潜在漏洞"""
    tech_stack = dspy.InputField(desc="技术栈信息")
    endpoints = dspy.InputField(desc="发现的API端点")
    vulnerabilities = dspy.OutputField(desc="漏洞列表JSON")
    risk_scores = dspy.OutputField(desc="风险评分")

# 2. 定义Module（执行策略）
analyzer = dspy.ChainOfThought(VulnerabilityAnalysis)

# 3. 自动优化（GEPA优化器）
optimizer = dspy.GEPA(metric=vuln_accuracy_metric)
optimized_analyzer = optimizer.compile(analyzer, trainset=labeled_examples)

# 4. 版本管理
optimized_analyzer.save("vuln_analyzer_v1.json")
```

**A/B测试框架**：

```
流量分桶（50%/50%）
  ├─ 变体A（当前prompt） → 指标采集（准确率/延迟/token消耗）
  └─ 变体B（优化prompt） → 指标采集
  ↓
统计显著性检验（p<0.05）
  ├─ B显著优于A → 自动推广B
  └─ 无显著差异 → 保持A，继续实验
```

## 七、关键技术决策总结

| 决策点 | 选型 | 理由 |
|-------|------|------|
| 推理框架 | vLLM | 通用高并发，TTFT最优，生态最广 |
| 量化方案 | AWQ INT4 | 质量损失1-3%，显存节省75%，生产默认 |
| LLM编排 | LangGraph + DSPy | LangGraph做Agent状态管理，DSPy做自动优化 |
| RAG架构 | Hybrid + Rerank + HyDE | 成本路由，按查询复杂度选择技术 |
| 训练方法 | 迭代DPO + GEPA | DPO资源低，GEPA用语言反思替代标量奖励 |
| 模型路由 | RouteLLM级联 | 削减75-85%成本，质量维持95% |
| Prompt管理 | DSPy save/load + A/B测试 | 版本控制 + 自动优化 + 统计验证 |
| 工具调用 | 并行 + 结构化返回 + 错误恢复 | 工具失败≠Agent失败 |
| 知识库 | Milvus + Neo4j | 向量检索 + 图关联查询混合 |
| 可观测性 | OpenTelemetry + ClickHouse | 全链路trace + 高性能分析 |

核心思路：**轻模型做实时决策（80%请求<1ms），重模型做深度分析（5%请求<5s），规则引擎兜底，GEPA持续优化Prompt，迭代DPO持续微调模型，形成自进化闭环**。
