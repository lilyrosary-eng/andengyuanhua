## 第十四轮：截图 bug3/bug4 真正根因修复 + wait-page 重写验证 (2026-07-10 18:30)
### bug3「截图卡顿 + 全灰遮罩」真正根因（前一轮仍没解决）
1. **二进制传输被 JSON 化**：`read_screenshot` 返回 `Vec<u8>`，Tauri v2 会序列化成 JSON 数字数组（`number[]`，每字节一个十进制数+逗号，比 base64 还大数倍）；前端 `new Blob([number[]])` 更会把数组 stringify 成乱码 → PNG 无法解码 → `<img>` 永远 load 不出 → `ready` 恒 false → 只剩 `bg-black/60` **全灰遮罩**。
   - 修复：Rust `read_screenshot` 改返回 `tauri::ipc::Response`（`Ok(tauri::ipc::Response::new(bytes))`）→ 前端 `invoke<ArrayBuffer>` 收到 ArrayBuffer → `new Blob([buf])` 正常解码。这是真正规避卡顿+灰屏的关键。
2. **DPI 坐标 × scale 是画蛇添足的错误**：查证权威结论（cn-sec「为什么你的截屏老是截不全」）——**DPI 感知进程（Tauri v2 默认 Per-Monitor-V2）中 `GetSystemMetrics`/`GetWindowRect` 返回的就是真实物理像素，与 GDI BitBlt 同坐标系，无需任何 ×scale**。前一轮误加的 `× scale` 在 150% 屏把 1920 当 2880 去截 → 超出屏幕 → 大片黑/灰、窗口矩形整体错位。
   - 修复 `screenshot.rs`：删除 `capture_screen`/`enum_callback`/`start_screenshot(vd)` 中的 `× scale`，直接用物理像素；**唯一保留** `create_overlay_window` 的 `÷ scale`（因 Tauri `inner_size/position` 以逻辑像素为单位，物理÷scale=逻辑，渲染后物理=逻辑×scale=整屏，WebView dpr=scale，前端 CSS×dpr↔物理 1:1 对齐）。

### bug4「窗口识别只能识别整屏」

## 第十九轮：截图卡死稳定性 bug 修复 (2026-07-10 21:52)
### 现象
用户反馈出现「卡死」。

### 根因（核心）
- `capture_window_full` 用 `PrintWindow(hwnd, hdc, 2 /*PW_RENDERFULLCONTENT*/)`。该 Win32 调用对**浏览器/Electron/UWP/硬件加速窗口**会**无限挂起**，Rust 命令线程被永久阻塞，前端 `await invoke('capture_window_full')` 永远不返回 → 覆盖窗卡在长截图选择态、无法完成也无法退出 = 卡死。

### 修复
1. **capture_window_full 加超时线程**：把 Win32 捕获拆到独立线程 `capture_window_full_inner`，主命令线程用 `mpsc::recv_timeout(5s)` 等待；超时即返回明确错误（不阻塞应用）。`screenshot.rs` 新增 `use std::sync::mpsc/thread/time::Duration`。
2. **PrintWindow flag 回退**：flag 2 返回 0（快速失败，非挂起）时回退 flag 0。
3. **start_screenshot 重复触发守卫**：覆盖窗已 `is_visible()` 时直接 `return Ok(())`，避免截图进行中再按热键重复触发。
4. **前端 captureLong catch 重置态**：失败/超时后 `setLongMode(false); setIsLong(false)`，覆盖窗退出长截图挑选态，不卡住。
5. **预览解码失败兜底**：`ScreenshotOverlay` 载入预览图的 `img.onerror` 直接 `onClose()`，避免永久白色 loading 假死。

### 验证
- `cargo check` 0 error；`npx tsc --noEmit` 0 error。需重新编译重启应用生效。
- **注意长期隐患**：超时线程若真被 PrintWindow 挂起，该线程会泄漏（无法 kill）；但已保证命令不阻塞、应用不卡死。后续若要做可靠长截图，需改用「滚动+BitBlt 拼接」或 `WM_PRINT` 方案替代 PrintWindow(2)。

## 第二十轮：彻底修复截图卡顿/偏移/长截图挂起 (2026-07-10 22:10)

### 根因 & 修复
1. **4-5s 卡顿（核心 UX）**：`list_windows` 用 `GetWindowTextW` 读每个窗口标题，会向目标线程发 `WM_GETTEXT`，目标窗口卡死时阻塞约 5s。`screenshot.rs` 改为 `SendMessageTimeoutW(WM_GETTEXT, SMTO_ABORTIFHUNG, 50ms)` 读标题，彻底消除卡顿。`EnumWindows` 自身 + `IsWindowVisible`/`GetWindowRect`/`GetClassNameW` 不发跨线程消息，安全。
2. **偏移/绿框不准**：`create_overlay_window` 用 `GetSystemMetrics`+除以 scale 定覆盖窗尺寸。`GetSystemMetrics` 在 Per-Monitor-DPI-Aware 进程里返回「系统 DPI 逻辑像素」，再除以 scale = 双重缩小 → 覆盖窗偏小、捕获区域与屏幕错位、绿框偏。改为新增 `virtual_desktop_rect()`：用 Win32 `EnumDisplayMonitors` 拿所有显示器**真实物理矩形并集**（回调 `monitor_enum_proc`，注意 rect 参数是 `*mut RECT`/LPRECT）。`start_screenshot` 每次截图前 `overlay.set_position(PhysicalPosition)`+`set_size(PhysicalSize)` 对齐该并集，捕获坐标直接取自并集（rx,ry,rw,rh），保证「捕获 == 显示」。`EnumDisplayMonitors` 的 lprcClip 用 `std::ptr::null()`（*const），hdc 用 `null_mut()`。
3. **长截图挂起（长期隐患，已根治）**：废掉 `PrintWindow(flag 2)`。新 `capture_window_full_inner` 用 `PrintWindow(flag 0)`（只截可见、绝不挂起）+ 程序化滚动：找带 `WS_VSCROLL` 的滚动目标（自身或 `EnumChildWindows` 子控件，`GetScrollInfo` 判范围），反复 `SendMessageW(WM_VSCROLL, SB_PAGEDOWN)` 翻页，`print_window_capture` 每页重截整窗，再用 `find_overlap`(重叠行比对)/`append_rows` 去重拼接；滚动到底或内容不再增加即停。非滚动窗口直接返回单屏。`capture_window_full` 仍用线程 + `recv_timeout(30s)` 守护（普通窗口 <1s，留足滚动时间）。辅助：`rows_match`/`find_overlap`/`append_rows`/`encode_png`。
4. **前端单击窗口放大偏移**：`ScreenshotOverlay` `onPointerUp` 单击窗口时原用 `win.x - ox`（物理）直接当 CSS 像素 → 放大 scale 倍偏移。改为 `(win.x - ox)/scale` 等换算成 CSS 像素再 `commitCrop`。绿框（hover）本就用 `(w.x-ox)/scale`，覆盖窗修好后即准。

### 踩坑（编译）
- `AppHandle` 在 Tauri v2 无 `monitors()` 方法 → 改用 `EnumDisplayMonitors`。
- winapi `SCROLLINFO.nPage/nMax` 等为 `u32`，与 i32 混算需显式 `as i32`；`SB_VERT` 是 `u32` 但 `GetScrollInfo` 的 nBar 要 `i32` → `SB_VERT as i32`。
- `MONITORENUMPROC` 第三参数类型是 `LPRECT` = `*mut RECT`（非 *const）→ 否则报 "types differ in mutability"。
- 移除原 `use winapi::um::winuser::{...}` 导入块（代码已全用全限定名），仅剩 `PrintWindow` 改全限定。
- `GetSystemMetrics` 兜底分支须包在 `unsafe {}`。

### 验证
- `cargo check` 0 error；`npx tsc --noEmit` 0 error。需 `tauri dev`/build 重新编译重启生效。
- 实测建议：① 普通/框选/单击窗口是否秒开且对齐 ② 绿框是否精准 ③ 长截图点「普通窗口」(记事本/设置)能滚拼出整页；点浏览器/VSCode 这类不响应 WM_VSCROLL 的窗口会快速返回「可见单屏」（不再 5s 卡死，但未必能拼出超长内容——属已知局限，因这类应用不用标准滚动条）。
