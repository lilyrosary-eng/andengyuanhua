## 第七轮：排查 notify "潜在隐患" (2026-07-10 16:43)
### 结论：第六轮诊断的 notify 隐患是误判，原配置本就正确
- 代码用 `notify::recommended_watcher`（main.rs:294）按编译目标选后端
- notify 8.2.0 的 Windows 后端（ReadDirectoryChangesWatcher）依赖 windows-sys，是 **非特性门控的硬性 target 依赖**（Cargo.toml:112），与 features 无关 → Windows 始终用原生后端，不回退 PollWatcher
- notify 8.2.0 无 "windows" 特性（可用特性见注册表），我之前加的 `features=["windows"]` 导致 cargo 版本选择失败
- macOS: `default=["macos_fsevent"]`，原配置显式 `macos_kqueue` 使 macOS 用 kqueue 而非 FSEvents（作者有意选择）
- 已还原 Cargo.toml 为 `default-features = false, features = ["macos_kqueue"]`，并补充后端选择机制注释
- 验证: cargo check 通过 EXIT:0
