- 用户反馈「关于」里点不了浅色/深色，「好像不是按钮」。根因：`bootPreview.ts` 原把「返回」按钮追加在 `#boot-preview` 覆盖层内，层级 z-index:2，被加载页 CSS 中多处 `position:fixed; inset:0; z-index:2` 的铺满元素（如 `body::after` 压暗、`.grid` 点阵）遮挡，点不到；且 `previewBootScreen` 开头 `if (getElementById(PREVIEW_ID)) return` 一旦预览没关掉就会一直留全屏覆盖层（z-index:99999）挡住后面所有点击，导致「关于」按钮也失效。
- 修复：①「返回」按钮改挂到 `document.body`，z-index 提到 2147483647（高于覆盖层 99999），`pointer-events:auto`，`onclick` 里 `e.stopPropagation()` + `cleanup()`，确保永远点得到；②去掉 early-return，改为进入时先调用上一次 `activeCleanup()`（清掉旧 overlay/按钮/计时器/key监听/注入 style）再重建，避免卡死残留；③覆盖层背景由 transparent 改为与加载页 body 一致的 `linear-gradient(160deg, var(--bg), var(--bg-2))`（兜底写死同色值），加 `overflow:hidden`，避免透出主程序背景；兜底 catch 分支同样把按钮挂到 body 且最高层级。
- 校验：eslint 0 错误；`tsc --noEmit` exit 0；read_lints 0。

## 启动白屏 + 预览仍点不了（第二轮修复）
- 用户反馈：①「关于」里还是点不了（返回/进入均失灵）；②打开软件仍白屏一会，要求把加载页优先级调到最高、内容可慢。
- 白屏根因：`index.html` 的 `#boot` 覆盖层靠异步 `fetch(waiting-page-*.html)` 后才注入内容，`fetch` 返回前覆盖层为空 → 一段空白/纯色屏。修复：创建 `#boot` 时**同步**写入极简加载界面（居中「岸灯鸢花」+ spinner），并把 `#boot` 的 `z-index` 提到 `2147483647`、`background` 同步设为对应主题渐变；`fetch` 命中后再替换为完整哥特页。这样首屏立即有加载界面，内容加载可慢。
- 「点不了」根因：旧版返回按钮在覆盖层内部（`z-index:2`）被加载页铺满元素压住关不掉 → 残留的 `#boot-preview`（`z-index:99999`）一直挡在全屏最上层，连「关于」按钮也点不了；上一轮只修了“下次能关”，没清已卡死的旧层。本轮：①`App.tsx` 挂载时调用 `clearStaleBootPreview()` 清理残留层；②`previewBootScreen` 进入时先强制移除已有层再重建（上一轮已做）；③返回按钮挂 `document.body`、`z-index` 最高、`pointer-events:auto`；④**新增兜底：点击预览任意处即关闭**（`overlay.addEventListener('click', cleanup)`），与按钮、Esc 三重保险，确保任何情况下都能退出预览。
- 校验：eslint 0 错误；`tsc --noEmit` exit 0。
- 备注：若白屏是 Tauri 原生窗口（HTML 加载前）闪白，那是 `src-tauri` 的 window `backgroundColor` 配置，需另调 tauri.conf，不在前端范围；当前修复已覆盖应用内空白期。

## 启动顺序重构 + 预览"点不了"根治（第三轮）
- 用户核心诉求：①「关于」里仍点不了（返回/进入失灵）；②开启软件先出现主题色空白、且极简加载在其后才出现；要求「开启后先只加载哥特加载界面 + 识别浅/深模式，加载界面加载完毕后再加载其它」。
- 根因定位：`loadPlugins` 用 `Promise.allSettled`，`onPluginsLoaded`/`__bootDone` 会正常触发，启动 `#boot` 在常态下会移除；但任一环节偏慢/异常时，`#boot`（`z-index:2147483647`）全屏渐变会残留盖住整个应用，且此时再开「关于」预览（99999）被启动层压住 → 表现为"整页点不了 / 预览关不掉"。主题空白来自：Tauri 原生窗口 `backgroundColor:#f5f5f0` + 旧代码在 `<body>` 末尾才创建 `#boot` 之间的空档。
- 修复 `index.html`：把哥特加载层的创建**移到 `<head>` 同步执行**，`appendChild` 到 `documentElement`（早于 `<body>` 绘制），立即铺同款渐变底色 → 首帧即为哥特风加载界面，**零主题空白**；内联线描莲花 + 品牌 + 进度条作为首屏，再 `fetch` 完整 `waiting-page-*.html` 无缝升级为完整哥特版；`__bootDone` 淡出时立即 `pointer-events:none`；**双保险兜底**：`window load` 与 **8s 硬超时** 强制收尾，任何异常都不会让全屏层永久遮挡点击。主题识别逻辑（localStorage.theme 优先，否则跟随系统）不变。
- 修复 `src/lib/bootPreview.ts`：「返回」按钮改为**与覆盖层同一同步调用里立即挂载**到 `document.body`（`z-index` 最高），不再等 fetch；保留 Esc + 点击任意处关闭三重保险；清理函数幂等可重复，统一移除覆盖层/按钮/注入样式/计时器。
- 校验：eslint 0 错误；`tsc --noEmit` exit 0。
- 结论：启动层现在必定自毁且永不挡点击；预览在任何情况下都能退出。若重新构建后仍"关于"按钮完全无反应（连预览都打不开），则需排查设置面板是否被其它固定层遮挡（非启动层问题），当前最可疑的遮挡源（启动层）已消除。

## 启动区哥特 CSS 污染导致"多模块变怪"（关键纠正）
- 用户反馈：重构启动顺序后"多个模块都变奇怪了"。根因定位在第三轮对 `index.html` 的改动：启动期 `fetch('waiting-page-*.html')` 把整页所有 `<style>` **永久 append 到 `document.head`**，且 `#boot` 移除后这些样式从不回收。
- `waiting-page-*.html` 的 CSS 含大量全局/通用类名选择器，注入 head 即**永久污染整个应用**：`body{font-family/color/background/overflow}`、`body::after`(固定 z-index:2 暗角遮罩)、`:root{--bg/--surface/--ink/--accent/--line...}` 覆盖应用自身变量、`.grid`(覆盖 Tailwind `display:grid` 工具类成 `position:fixed;inset:0`)、`.main/.status/.progress/.heading/.footer/.brand` 等撞名 → 各模块布局与字体崩坏。
- 为何 dev 出现、构建日志没报：dev 下 Vite 能服务 `tools/` 文件，`fetch` 成功→污染；生产构建 `fetch` 404→走 `.catch` 内联兜底→不污染（所以 deploy/build 看似正常，是假象）。
- 修复：`index.html` 启动期**移除**对完整等待页的 fetch 与全局样式注入，只保留已内置的线描莲花+进度条哥特首屏（同样满足"先加载哥特界面+识别深浅模式+加载完再加载其它"，且零污染）；完整哥特页仅保留在「关于」预览（bootPreview）按需注入并随关闭回收。进度钩子（main/PluginHost/App 的 `__bootProgress`/`__bootDone`）照常驱动内联加载层，加载完自动淡出。
- 教训：**绝不要把第三方整页 HTML 的 `<style>` 原样塞进主应用 `<head>`**——其全局/通用类名选择器会污染全站。预览场景必须隔离（Shadow DOM 或关闭时严格回收）。之前 93 行"已核查主应用未用这些类名、无污染"的判断对"关于预览"成立，但对"启动期常驻注入"不成立，特此纠正。
